# Web Design Enhancer V3 — Plan directeur d’implémentation

## 1. Vision du produit

### 1.1 Mission

Transformer `web-design-enhancer` en un système de conception et de validation d’interfaces **piloté par preuves**, indépendant du modèle, capable de guider puis contrôler des agents tels que Claude Code, Codex/GPT, Cursor, Langdock, Grok, OpenCode et Antigravity.

V3 ne doit plus seulement demander au modèle de respecter un workflow. Elle doit :

1. enregistrer l’état réel du projet ;
2. déterminer l’étape autorisée ;
3. exécuter les contrôles hors du modèle ;
4. invalider les preuves après une modification du code ;
5. empêcher une validation finale sans artefacts frais ;
6. distinguer les règles absolues, contractuelles et heuristiques ;
7. évaluer l’UX, le contenu, l’accessibilité, le responsive, la performance et la qualité visuelle ;
8. mesurer le comportement des différents modèles avec un benchmark reproductible.

### 1.2 Positionnement

V3 n’est pas un « méga-prompt de web design ». C’est un **orchestrateur de qualité pour agents de conception et de développement frontend**.

Il combine quatre fonctions :

- **Direction** : transformer une demande en intention, architecture et contrat de conception.
- **Orchestration** : imposer un ordre d’exécution avec une machine à états.
- **Validation** : contrôler mécaniquement et visuellement le résultat.
- **Évaluation** : mesurer la conformité des modèles et la qualité des sorties.

### 1.3 Utilisateur principal

Développeur solo assisté par IA, travaillant sur des sites marketing ou des produits web, et souhaitant obtenir une qualité reproductible sans surveiller manuellement chaque décision de l’agent.

### 1.4 Principe fondamental

> Le modèle propose et implémente. L’orchestrateur autorise, vérifie et bloque.

Aucune affirmation du modèle — « gate passé », « responsive vérifié », « rendu validé » — ne constitue une preuve. Une preuve doit provenir d’une commande, d’un artefact, d’un hash ou d’un évaluateur identifié.

## 2. Objectifs et critères de réussite

### 2.1 Objectifs de V3

- Conserver les meilleurs éléments de V2 : détection anti-slop, contrat `DESIGN.md`, audits, Playwright, boucle de correction et tests.
- Remplacer le workflow textuel fragile par une machine à états persistante.
- Rendre le noyau indépendant de Claude Code ou de tout autre agent.
- Ajouter les domaines insuffisamment couverts : architecture de l’information, UX fonctionnelle, contenu, formulaires, responsive, accessibilité comportementale, performance, internationalisation et maintenabilité.
- Transformer les archétypes visuels en références modulaires plutôt qu’en templates obligatoires.
- Créer des adaptateurs légers pour chaque environnement agentique.
- Construire un benchmark multi-modèles et un corpus de tâches de référence.

### 2.2 Critères de réussite produit

V3 est considérée comme utilisable lorsque :

- un projet peut être initialisé et conduit jusqu’à la livraison avec une commande principale ;
- l’état du workflow survit à l’arrêt d’une session agentique ;
- une modification du code invalide automatiquement les validations concernées ;
- aucun statut `passed` ne peut être écrit directement par le modèle sans preuve reconnue ;
- les modes sans navigateur ou sans MCP sont explicitement dégradés ;
- au moins trois environnements sont supportés de bout en bout au lancement ;
- les mêmes contrats et résultats sont lisibles par tous les adaptateurs ;
- le benchmark permet de comparer plusieurs modèles sur la conformité au workflow ;
- les faux positifs des règles anti-template sont mesurés et documentés ;
- un humain peut comprendre la raison exacte d’un blocage et l’action nécessaire.

### 2.3 Non-objectifs initiaux

Pour garder une portée réaliste pour un développeur solo, V3.0 ne cherchera pas immédiatement à :

- remplacer Figma ou produire des fichiers Figma natifs ;
- devenir un constructeur de sites complet ;
- héberger un service SaaS multi-utilisateur ;
- entraîner un modèle visuel propriétaire ;
- supporter tous les frameworks frontend dès la première version ;
- garantir objectivement la « beauté », notion qui restera évaluée par dimensions et preuves ;
- automatiser une validation humaine réelle lorsque personne n’est disponible.

## 3. Principes d’architecture

### 3.1 Noyau indépendant du modèle

Le noyau ne doit contenir aucune hypothèse propre à Claude, GPT, Grok ou Cursor. Il expose :

- un CLI ;
- des fichiers d’état JSON ;
- des schémas JSON Schema ;
- des codes de sortie stables ;
- des commandes déterministes ;
- des rapports lisibles par machine et par humain.

Les environnements agents utilisent des adaptateurs, mais ne modifient pas les règles du noyau.

### 3.2 Progression par états

Le système représente le projet comme une machine à états :

```text
UNINITIALIZED
    ↓
INTENT_REQUIRED
    ↓
INTENT_VALIDATED
    ↓
RESEARCH_REQUIRED
    ↓
RESEARCH_VALIDATED
    ↓
ARCHITECTURE_REQUIRED
    ↓
ARCHITECTURE_VALIDATED
    ↓
CONTRACT_REQUIRED
    ↓
CONTRACT_VALIDATED
    ↓
IMPLEMENTATION_ALLOWED
    ↓
IMPLEMENTATION_DIRTY
    ↓
MECHANICAL_REVIEW_REQUIRED
    ↓
VISUAL_REVIEW_REQUIRED
    ↓
INDEPENDENT_REVIEW_REQUIRED
    ↓
READY_TO_DELIVER
```

Un état ne change que si une transition connue réussit. Le modèle ne peut pas écrire directement `READY_TO_DELIVER`.

### 3.3 Validation fondée sur les preuves

Chaque contrôle produit une enveloppe de preuve :

```json
{
  "schema_version": "3.0",
  "check_id": "responsive.visual",
  "status": "passed",
  "executed_at": "2026-07-10T14:00:00Z",
  "executor": "wde-core",
  "tool_version": "3.0.0",
  "source_hash": "sha256:...",
  "contract_hash": "sha256:...",
  "environment": {
    "url": "http://localhost:3000",
    "browser": "chromium",
    "viewport": "375x812"
  },
  "artifacts": [
    "audit-results/responsive/375/home.png",
    "audit-results/responsive/report.json"
  ],
  "result_digest": "sha256:..."
}
```

### 3.4 Invalidation par dépendances

Une modification ne doit pas invalider arbitrairement tout le projet. V3 construit un graphe de dépendances :

- changement du brief → invalide architecture, contrat et validations suivantes ;
- changement de `DESIGN.md` → invalide verrou structurel, dérive, audit visuel et review ;
- changement CSS/JSX → invalide audits code, rendu, accessibilité, responsive, performance et verdict final ;
- changement de texte uniquement → invalide contenu, layout, accessibilité et captures concernées ;
- changement de configuration de build → invalide build, performance et rendu.

### 3.5 Règles contextualisées

Toutes les règles appartiennent à l’une des catégories suivantes :

1. **Safety / Truth** — absolues : pas de faux témoignages, faux clients ou fausses métriques.
2. **Accessibility Critical** — bloquantes : clavier inaccessible, contraste critique, absence de label, etc.
3. **User Requirement** — bloquantes lorsque clairement formulées.
4. **Brand Contract** — bloquantes lorsque présentes dans le contrat validé.
5. **Functional Quality** — généralement bloquantes selon la sévérité.
6. **Anti-template Heuristic** — score ou avertissement, sauf combinaison grave.
7. **Taste Preference** — conseil uniquement.

Une règle doit toujours indiquer sa catégorie, sa sévérité, sa preuve et la correction recommandée.

### 4.2 Choix technologiques

Pour préserver les investissements existants, conserver **Python** pour le noyau.

Recommandations :

- Python 3.11+ ;
- `pydantic` ou JSON Schema pour les contrats ;
- `typer` pour le CLI ;
- `pytest` pour les tests ;
- Playwright Python ou Node pour les contrôles navigateur ;
- `networkx` facultatif pour le graphe de dépendances ;
- SQLite facultatif en V3.1, mais fichiers JSON en V3.0 pour la transparence ;
- rapports HTML statiques pour consultation locale ;
- JSON Lines pour les journaux d’exécution.

Éviter de réécrire immédiatement tous les scripts V2. Les encapsuler derrière une interface de check, puis les migrer progressivement.

## 5. Contrats et fichiers du projet utilisateur

### 5.1 Répertoire de contrôle

Chaque projet géré contient :

```text
.wde/
├── project.json
├── state.json
├── capabilities.json
├── evidence/
├── reports/
├── logs/
└── cache/
CREATIVE-BRIEF.md
EXPERIENCE-CONTRACT.md
DESIGN.md
STRUCTURAL-LOCK.md
```

### 5.2 `project.json`

Contient la configuration stable :

- identifiant du projet ;
- type de produit ;
- framework ;
- commandes de développement, build et test ;
- URL locale ;
- chemins source ;
- pages critiques ;
- profils d’audit ;
- politique active ;
- adaptateur utilisé ;
- capacités disponibles.

### 5.3 `state.json`

Contient uniquement l’état piloté par le noyau :

- phase actuelle ;
- transitions réussies ;
- contrôles valides ;
- contrôles invalidés ;
- hashes ;
- blocages ;
- prochaine action autorisée ;
- historique minimal.

Le fichier est réécrit par le CLI. Une signature HMAC locale est facultative ; le plus important est que le CLI recalcule toutes les preuves avant d’autoriser la livraison.

### 5.4 `CREATIVE-BRIEF.md`

Le brief doit inclure :

- sujet concret ;
- public principal ;
- action principale attendue ;
- problème utilisateur ;
- émotion recherchée ;
- éléments à préserver ;
- éléments explicitement interdits ;
- pari créatif principal ;
- dimension héroïque dominante ;
- règle rompue et justification ;
- références et anti-références ;
- contraintes de marque ;
- contraintes de contenu ;
- niveau de risque créatif accepté.

Le fichier doit posséder un bloc de métadonnées lisible par machine ou un compagnon JSON généré automatiquement.

### 5.5 `EXPERIENCE-CONTRACT.md`

Nouveau contrat consacré à l’UX :

- objectifs par page ;
- architecture de l’information ;
- parcours critiques ;
- navigation ;
- actions principales et secondaires ;
- états vide, chargement, erreur, succès et permission ;
- formulaires ;
- comportements responsive ;
- exigences d’accessibilité ;
- contenu réel ou statut des données ;
- critères d’acceptation fonctionnels.

### 5.6 `DESIGN.md`

Le contrat visuel reste présent, mais il doit être restructuré autour de décisions spécifiques :

- palette nommée ;
- rôles typographiques ;
- échelle ;
- densité ;
- grille ;
- composition ;
- vocabulaire de formes ;
- traitement des images ;
- mouvement ;
- états interactifs ;
- signature visuelle ;
- règles de cohérence ;
- exceptions justifiées.

Les archétypes deviennent des références facultatives. Le contrat ne doit pas se réduire à « archetype 06 + tokens par défaut ».

### 5.7 `STRUCTURAL-LOCK.md`

Il verrouille les décisions coûteuses à changer :

- structure des pages ;
- hiérarchie des sections ;
- navigation ;
- signature du hero ;
- modèle responsive ;
- primitives de mise en page ;
- composants principaux ;
- stratégie de contenu ;
- stratégie d’images et d’animations.

## 6. Modules fonctionnels à construire

## 6.1 Module Intent & Research

### Responsabilités

- valider la complétude du brief ;
- distinguer faits, hypothèses et inventions ;
- enregistrer les références ;
- détecter les contradictions ;
- produire les critères d’acceptation initiaux.

### Contrôles

- objectif principal unique ou priorisé ;
- public identifiable ;
- contenu manquant explicite ;
- références accessibles ou documentées ;
- interdiction des données fictives présentées comme réelles.

### Sortie

`intent-report.json` avec erreurs, avertissements et questions non résolues.

## 6.2 Module Information Architecture

### Responsabilités

- inventaire des contenus ;
- sitemap ;
- hiérarchie des pages ;
- séquence de lecture ;
- navigation globale et locale ;
- priorité mobile.

### Contrôles

- chaque section sert un objectif ;
- absence de sections automatiques sans justification ;
- CTA cohérents ;
- profondeur de navigation raisonnable ;
- titres uniques et hiérarchisés ;
- ordre mobile défini.

### Sorties

- `content-inventory.json` ;
- `sitemap.json` ;
- `page-objectives.json` ;
- section IA de `EXPERIENCE-CONTRACT.md`.

## 6.3 Module Interaction & State Coverage

### Responsabilités

- définir les actions ;
- modéliser les parcours ;
- couvrir les états ;
- contrôler les actions destructrices ;
- vérifier feedback et réversibilité.

### États minimums

- initial ;
- chargement ;
- vide ;
- partiel ;
- erreur récupérable ;
- erreur bloquante ;
- succès ;
- non autorisé ;
- hors ligne si pertinent ;
- contenu extrême.

### Contrôles navigateur

- tous les contrôles déclenchent un comportement ;
- les boutons ne sont pas décoratifs ;
- les retours utilisateur sont visibles et compréhensibles ;
- les modales rendent le focus correctement ;
- les actions critiques demandent confirmation ou offrent une annulation.

## 6.4 Module Content Design

### Responsabilités

- terminologie ;
- proposition de valeur ;
- libellés ;
- messages d’erreur ;
- écrans vides ;
- cohérence des actions ;
- preuve des claims.

### Règles

- langage concret ;
- verbes d’action stables ;
- pas de jargon sans nécessité ;
- aucune métrique inventée ;
- aucun faux logo ou témoignage ;
- les erreurs disent ce qui s’est passé et comment corriger ;
- les placeholders ne remplacent pas les labels.

### Audit de robustesse

Tester avec :

- titres longs ;
- noms longs ;
- absence d’image ;
- paragraphes deux fois plus longs ;
- nombres importants ;
- données nulles.

## 6.5 Module Visual Direction

### Évolution des archétypes

Conserver les dix archétypes comme bibliothèque historique, mais sélectionner une direction par axes :

- densité ;
- contraste ;
- géométrie ;
- matérialité ;
- expressivité typographique ;
- intensité chromatique ;
- régularité de grille ;
- intensité de mouvement ;
- rapport contenu/décoration ;
- temporalité culturelle.

Chaque axe reçoit :

- une valeur de 0 à 10 ;
- une justification ;
- une preuve dans le rendu ;
- une relation avec le brief.

### Signature obligatoire

Une seule signature mémorable doit être déclarée :

- interaction ;
- composition ;
- typographie ;
- illustration ;
- visualisation ;
- mouvement ;
- objet 3D ;
- navigation.

Le système vérifie son existence sans exiger plusieurs artifices décoratifs.

### Anti-template contextualisé

Le détecteur attribue :

- une probabilité de pattern générique ;
- le contexte qui le rend suspect ;
- les éléments associés ;
- une recommandation ;
- une exception possible si le contrat le justifie.

Un gradient ou une grille de trois éléments ne bloque plus isolément. Une combinaison non justifiée peut bloquer.

## 6.6 Module Responsive Strategy

### Avant implémentation

Définir pour chaque page :

- priorité de contenu ;
- comportement de navigation ;
- reflow des grilles ;
- traitement des tableaux ;
- images ;
- ordre de lecture ;
- actions persistantes ;
- suppression ou simplification de mouvement.

### Viewports minimums

- 320 × 568 ;
- 375 × 812 ;
- 768 × 1024 ;
- 1024 × 768 ;
- 1440 × 900 ;
- 1920 × 1080 pour détection de largeur excessive.

Le profil peut adapter cette liste selon le produit.

### Tests

- overflow horizontal ;
- collisions ;
- texte tronqué ;
- cible tactile ;
- visibilité du focus ;
- navigation ;
- tableaux ;
- modales ;
- orientation paysage ;
- zoom 200 % ;
- robustesse du contenu.

## 6.7 Module Accessibility

### Audits automatiques

- axe-core si disponible ;
- structure sémantique ;
- titres ;
- labels ;
- noms accessibles ;
- contraste ;
- alt ;
- focus ;
- zones tactiles ;
- réduction du mouvement ;
- zoom et reflow.

### Scénarios comportementaux

- navigation complète au clavier ;
- ouverture et fermeture des menus ;
- piège et retour du focus dans les modales ;
- soumission de formulaire en erreur ;
- annonce des mises à jour dynamiques ;
- navigation au lecteur d’écran, si l’environnement le permet.

### Politique de sévérité

- critique : bloque ;
- majeure : bloque sauf dérogation documentée ;
- mineure : avertissement avec dette ;
- bonne pratique : recommandation.

## 6.8 Module Forms

### Couverture

- labels et descriptions ;
- formats attendus ;
- validation ;
- erreurs par champ et globale ;
- conservation des données ;
- autocomplétion ;
- clavier mobile ;
- soumission multiple ;
- chargement ;
- succès ;
- action destructive ;
- confidentialité.

### Tests automatiques

Chaque formulaire critique reçoit un scénario Playwright :

1. soumission vide ;
2. valeurs invalides ;
3. correction ;
4. soumission valide ;
5. vérification du focus et des messages.

## 6.9 Module Performance

### Budgets configurables

Profil marketing par défaut :

- JavaScript initial compressé ≤ 250 Ko, hors exception justifiée ;
- images hero optimisées et dimensionnées ;
- CLS ≤ 0,1 ;
- absence de longues tâches évitables ;
- nombre de fontes et graisses limité ;
- LCP local mesuré et suivi, sans prétendre reproduire le terrain.

Profil application : budgets adaptés aux contraintes fonctionnelles.

### Contrôles

- poids des bundles ;
- images non optimisées ;
- polices ;
- vidéos ;
- scripts tiers ;
- animations coûteuses ;
- rendu hors écran ;
- chargement paresseux ;
- stabilité de layout.

Les chiffres locaux doivent être présentés comme mesures de laboratoire, pas comme données réelles utilisateur.

## 6.10 Module Localization Resilience

### Tests pseudo-locale

- expansion du texte à 30–50 % ;
- caractères accentués ;
- chaînes RTL ;
- nombres et devises ;
- dates longues ;
- noms de personnes longs.

### Contrôles

- aucune concaténation de phrase fragile ;
- mise en page compatible RTL lorsque demandée ;
- composants flexibles ;
- textes non intégrés aux images ;
- attributs `lang` et `dir` corrects.

## 6.11 Module Maintainability

### Contrôles

- duplication de tokens ;
- couleurs et espacements magiques répétés ;
- composants trop volumineux ;
- variantes incohérentes ;
- styles morts ;
- dépendances inutiles ;
- séparation primitives/composants métier ;
- erreurs console ;
- TypeScript ou lint selon le projet.

Ce module ne doit pas imposer une architecture universelle. Il contrôle principalement la cohérence avec les conventions déclarées du projet.

## 7. Moteur de politiques et système de checks

### 7.1 Interface d’un check

Chaque contrôle implémente :

```python
class Check:
    id: str
    version: str
    dependencies: list[str]
    required_capabilities: list[str]

    def applicable(self, context) -> bool: ...
    def run(self, context) -> CheckResult: ...
    def evidence(self, result) -> EvidenceEnvelope: ...
```

### 7.2 Résultat standard

```json
{
  "check_id": "content.truthfulness",
  "status": "failed",
  "severity": "blocking",
  "category": "safety_truth",
  "summary": "Trois métriques ne disposent d’aucune source.",
  "findings": [
    {
      "rule_id": "TRUTH-003",
      "location": "src/pages/Home.tsx:84",
      "evidence": "Join 50,000 teams",
      "reason": "Aucune donnée source déclarée.",
      "fix": "Fournir une source ou reformuler sans chiffre."
    }
  ]
}
```

### 7.3 Profils de politiques

Prévoir des profils composables :

- `web-base` ;
- `marketing-site` ;
- `product-app` ;
- `dashboard` ;
- `editorial` ;
- `ecommerce` ;
- `portfolio` ;
- `native-mobile` ultérieurement.

Un profil sélectionne les checks, seuils et scénarios. Il ne choisit pas automatiquement une esthétique.

### 7.4 Dérogations

Une dérogation doit contenir :

- règle concernée ;
- justification ;
- auteur ;
- date ;
- durée ou condition d’expiration ;
- risque accepté.

Les dérogations aux règles de vérité ou aux problèmes critiques d’accessibilité ne sont pas autorisées par défaut.

## 8. Inspection visuelle et évaluation indépendante

### 8.1 Pipeline visuel

1. démarrer le serveur avec une commande déclarée ;
2. vérifier que l’URL répond ;
3. capturer les pages critiques à chaque viewport ;
4. collecter DOM, accessibilité, console et réseau ;
5. effectuer les interactions principales ;
6. générer les diffs et métriques ;
7. soumettre un paquet d’évaluation à un juge distinct ;
8. consolider verdict mécanique et verdict visuel.

### 8.2 Paquet remis au juge

Le juge reçoit :

- brief condensé ;
- contrat UX ;
- contrat visuel ;
- captures ;
- objectifs de chaque page ;
- résultats mécaniques pertinents ;
- grille d’évaluation structurée.

Il ne reçoit pas :

- les auto-justifications du modèle constructeur ;
- un score esthétique prérempli ;
- une instruction lui demandant de confirmer la réussite.

### 8.3 Dimensions du verdict

- fidélité à l’intention ;
- clarté de la hiérarchie ;
- cohérence visuelle ;
- qualité typographique ;
- composition ;
- qualité responsive ;
- crédibilité du contenu ;
- présence et exécution de la signature ;
- impression de template générique ;
- problèmes visibles précis.

Chaque dimension exige une preuve localisée. Le juge ne peut pas retourner uniquement une note.

### 8.4 Modes d’indépendance

- **Fort** : modèle ou humain distinct, contexte séparé.
- **Moyen** : même fournisseur mais nouvelle session sans historique de construction.
- **Faible** : même agent, nouvelle passe structurée.
- **Indisponible** : aucun juge ; livraison marquée `unverified`, jamais présentée comme revue indépendamment.

## 9. Adaptateurs multi-agents

### 9.1 Contrat de capacités

Chaque adaptateur déclare :

```json
{
  "adapter": "claude-code",
  "capabilities": {
    "shell": true,
    "filesystem": true,
    "browser": true,
    "vision": true,
    "subagents": true,
    "hooks": true,
    "final_response_interception": false
  }
}
```

Le noyau calcule ensuite le mode : complet, partiel ou dégradé.

### 9.2 Adaptateur générique

À construire en premier. Il doit fonctionner avec tout agent capable de :

- lire des fichiers ;
- exécuter des commandes ;
- modifier le projet.

Il fournit un fichier d’instructions minimal :

1. exécuter `wde status --json` ;
2. effectuer uniquement l’action indiquée par `next_action` ;
3. exécuter `wde transition ...` ;
4. ne jamais écrire dans `.wde/evidence` ;
5. avant livraison, exécuter `wde deliver-check`.

### 9.3 Claude Code

- `SKILL.md` mince ;
- commandes explicites ;
- hooks si disponibles pour `post-write` et `pre-delivery` ;
- sous-agent séparé pour la review visuelle lorsque possible.

### 9.4 Codex / GPT

- instructions de dépôt ;
- CLI comme source de vérité ;
- commandes courtes et sorties JSON ;
- éviter de dépendre d’un format propriétaire de skill.

### 9.5 Cursor

- règles projet minces ;
- tâches ou commandes enregistrées ;
- workflow utilisable même sans interception de la réponse finale ;
- rapport explicite lorsque la livraison ne peut pas être techniquement bloquée.

### 9.6 Langdock

Créer un skill textuel concise qui orchestre le CLI. Ne pas dupliquer les longues références dans les instructions. Pour une distribution complète, fournir un paquet contenant scripts, références et templates.

### 9.7 Grok, OpenCode et Antigravity

Commencer avec l’adaptateur générique. Ajouter un adaptateur dédié seulement si l’environnement fournit :

- hooks ;
- sous-agents ;
- format de skill ;
- outils navigateur ;
- interception des réponses ;
- gestion de permissions.

Chaque adaptateur doit avoir des tests de contrat identiques.

## 11. Migration de V2 vers V3

### 11.1 Inventaire

Classer chaque composant V2 :

- conserver tel quel ;
- encapsuler ;
- refactoriser ;
- remplacer ;
- supprimer.

### 11.2 Éléments à conserver et encapsuler

- détecteur anti-slop ;
- audit d’accessibilité ;
- audit des espacements ;
- audit de composition ;
- audit layout ;
- capture Playwright ;
- boucle de raffinement ;
- corpus de calibration ;
- tests de régression utiles.

### 11.3 Éléments à refactoriser

- `check.py` devient un registre de checks et un planificateur ;
- `SKILL.md` devient un adaptateur mince ;
- les gates G/F deviennent des IDs de politiques stables ;
- les artefacts Eyes adoptent le schéma de preuve commun ;
- les archétypes deviennent des références et axes ;
- le verdict esthétique adopte les niveaux d’indépendance.

### 11.4 Compatibilité

Créer une commande :

```bash
wde migrate-v2
```

Elle :

- détecte les fichiers V2 ;
- crée `.wde/project.json` ;
- importe les contrats ;
- convertit les résultats connus en preuves historiques non fiables ;
- demande de relancer les checks nécessaires ;
- ne marque jamais automatiquement un projet comme prêt à livrer.

## 12. Benchmark multi-modèles

### 12.1 Objectif

Mesurer deux dimensions séparées :

1. **Conformité procédurale** : le modèle suit-il le workflow ?
2. **Qualité du résultat** : le résultat est-il utile, robuste et distinctif ?

### 12.2 Corpus initial

Créer 12 tâches :

- landing B2B technique ;
- marque cosmétique ;
- média éditorial ;
- portfolio créatif ;
- dashboard financier ;
- formulaire d’inscription complexe ;
- application avec états vides et erreurs ;
- site multilingue ;
- e-commerce mobile ;
- redesign avec identité existante ;
- brief volontairement vague ;
- brief contradictoire.

Chaque tâche fournit :

- demande utilisateur ;
- assets ;
- données autorisées ;
- contraintes ;
- critères cachés ;
- scénario de test ;
- problèmes que le modèle doit détecter.

### 12.3 Mesures procédurales

- activation correcte ;
- lecture des entrées nécessaires ;
- absence de code avant autorisation ;
- commandes réellement exécutées ;
- absence de preuves fabriquées ;
- relance des checks après modification ;
- respect des blocages ;
- transparence en mode dégradé.

### 12.4 Mesures qualité

- succès des parcours ;
- accessibilité ;
- responsive ;
- performance ;
- robustesse du contenu ;
- fidélité au brief ;
- distinction visuelle ;
- préférence humaine par comparaison ;
- nombre d’itérations ;
- coût et temps.

### 12.5 Comparaison

Pour chaque modèle/environnement :

- exécuter au moins trois runs par tâche ;
- conserver version et paramètres ;
- utiliser les mêmes assets ;
- séparer constructeur et juge ;
- publier médiane, variance et échecs ;
- éviter un classement global simpliste ;
- présenter les forces selon le type de tâche.

## 13. Stratégie de tests

### 13.1 Tests unitaires

- transitions d’état ;
- validation de schémas ;
- hashing ;
- invalidation ;
- politiques ;
- dérogations ;
- normalisation des résultats ;
- calcul de sévérité.

### 13.2 Tests d’intégration

- wrapper d’un script V2 ;
- génération d’une preuve ;
- modification du code et invalidation ;
- démarrage serveur ;
- capture navigateur ;
- rapport consolidé ;
- détection des capacités.

### 13.3 Tests end-to-end

Scénario minimum :

1. initialisation ;
2. validation brief ;
3. contrat UX ;
4. contrat visuel ;
5. autorisation d’implémentation ;
6. modification d’un fixture frontend ;
7. audits ;
8. échec volontaire ;
9. correction ;
10. nouvelle preuve ;
11. livraison autorisée.

### 13.4 Tests anti-contournement

- édition manuelle de `state.json` ;
- copie d’une ancienne preuve ;
- preuve avec mauvais hash ;
- capture antérieure au code ;
- verdict `self` présenté comme externe ;
- URL inaccessible ;
- rapport vide ;
- commande interrompue ;
- check non applicable présenté comme passé ;
- changement de fuseau ou horloge.

### 13.5 Tests des adaptateurs

Tous les adaptateurs passent un contrat :

- savent obtenir l’état ;
- savent identifier la prochaine action ;
- ne modifient pas les preuves ;
- signalent leurs capacités ;
- exposent le mode dégradé ;
- utilisent `deliver-check`.

## 14. Roadmap d’implémentation pour un développeur solo

## Phase 0 — Cadrage et gel de V2

**Durée indicative : 2 à 3 jours**

### Tâches

- créer une branche `v3` ;
- taguer la dernière version V2 stable ;
- inventorier scripts et tests ;
- lister les contrats JSON existants ;
- identifier les dépendances implicites ;
- définir les objectifs mesurables de V3.0 ;
- sélectionner trois environnements de lancement : générique, Claude Code et Codex/GPT.

### Sortie

- matrice migration ;
- liste des checks V2 ;
- ADR sur Python et fichiers JSON ;
- backlog priorisé.

### Critère de fin

Aucun développement majeur ne commence sans savoir quelles briques V2 sont réutilisées.

## Phase 1 — Noyau d’état et preuves

**Durée indicative : 1 à 2 semaines**

### Tâches

- créer le package `wde` ;
- implémenter `project.json` et `state.json` ;
- créer les schémas ;
- implémenter hashing ;
- créer la machine à états ;
- créer transitions ;
- implémenter enveloppes de preuve ;

- protéger les transitions ;
- ajouter journaux JSONL.

### Tests

- transitions valides et invalides ;
- état corrompu ;
- preuve invalide ;
- source hash modifié ;
- reprise après interruption.

### Critère de fin

Un fixture peut progresser jusqu’à `IMPLEMENTATION_ALLOWED`, puis être invalidé après modification.

## Phase 2 — Registre de checks et compatibilité V2

**Durée indicative : 1 à 2 semaines**

### Tâches

- définir l’interface `Check` ;
- créer registre et plan d’exécution ;

- normaliser leurs sorties ;

4. anti-template ;
5. layout/spacing.

Les scripts V2 peuvent être appelés par le nouveau noyau sans perdre leurs tests existants.

## 17. Risques et réponses

### Risque : portée trop grande pour une personne

**Réponse** : livrer d’abord le noyau d’orchestration, encapsuler les scripts V2, limiter V3.0 à trois adaptateurs et quatre tâches benchmark.

### Risque : multiplication des faux positifs

**Réponse** : distinguer blocage et heuristique, fournir exemptions, constituer corpus de calibration, suivre précision et rappel.

### Risque : les agents ignorent encore le CLI

**Réponse** : instructions très courtes, `wde next --json`, hooks lorsqu’ils existent, CI facultative, et benchmark procédural. Ne jamais promettre un blocage que l’environnement ne permet pas.

### Risque : review esthétique instable

**Réponse** : critères dimensionnels, preuves obligatoires, comparaison à des ancres, répétitions, variance et niveau d’indépendance visible.

### Risque : V3 devient bureaucratique

**Réponse** : profils, étapes automatiques, contrats courts, génération assistée et mode rapide pour petits projets. Chaque gate doit démontrer sa valeur.

### Risque : les axes visuels produisent encore des designs génériques

**Réponse** : exiger une signature liée au sujet, une anti-référence, une justification de composition et un contrôle comparatif entre projets du corpus.

### Risque : dépendance excessive à Playwright ou au réseau

**Réponse** : détection de capacités, modes dégradés explicites, audits statiques de secours et aucune preuve visuelle fictive.

## 18. Definition of Done de V3.0

V3.0 est terminée si :

- installation propre réussie ;
- `wde init` crée un projet valide ;
- `wde status` retourne un état fiable ;
- une transition non autorisée échoue ;
- les contrats intention, UX et design sont validés ;
- le code ne peut être déclaré livrable sans audits frais ;
- les preuves comportent hashes, timestamps et artefacts ;
- les changements invalident les preuves dépendantes ;
- les principaux audits V2 sont intégrés ;
- Playwright teste plusieurs viewports et un parcours critique ;
- le rapport sépare erreurs, avertissements et préférences ;
- la review déclare son niveau d’indépendance ;
- le mode dégradé est visible ;
- trois adaptateurs passent les tests de contrat ;
- quatre tâches benchmark sont reproductibles ;
- la documentation permet à un développeur solo de lancer le workflow ;
- aucune promesse « non contournable » n’est faite pour un environnement sans mécanisme réel de blocage.

## 20. Décision finale recommandée

Construire V3 comme un **noyau d’orchestration déterministe avec plugins de validation**, et non comme un unique skill plus long.

Le skill doit devenir une fine couche d’orientation :

- détecter que V3 s’applique ;
- exécuter `wde status --json` ;
- suivre `next_action` ;
- charger uniquement la référence nécessaire ;
- appeler `wde deliver-check` avant toute livraison.

Tout ce qui exige de la fiabilité — état, ordre, preuves, hashes, invalidation, seuils et blocages — doit vivre dans le code. Tout ce qui exige du jugement — intention, composition, contenu et critique — peut être confié au modèle, mais sous contrat et avec évaluation séparée.

C’est cette séparation qui fera passer le projet d’un excellent ensemble d’instructions à une véritable infrastructure de qualité pour agents frontend.
